# HARMONY — Pillar 2: Entity Type Definitions
## Central Coast MVP Entity Schemas
**Date:** April 19, 2026
**Version:** 1.0
**Applies to:** Pillar 2 Milestones 4–7
**Authority:** These schemas define the attribute mappings that the Entity Extractor (Task 6) uses to convert source features into Harmony entities. Attribute names in the `harmony_field` column are authoritative — source-specific field names are mapped to these during extraction.

---

## Schema Conventions

All entity types share a common base schema inherited from the Pillar 1 entity model:

| Base Field | Type | Source | Notes |
|---|---|---|---|
| `entity_id` | string | Generated by Pillar 1 API | Canonical entity identifier, assigned at registration |
| `entity_type` | string | Set by extractor from manifest | One of: `zoning_area`, `cadastral_lot`, `building`, `road_segment` |
| `primary_cell_id` | string | Computed by cell assignment engine | Cell key of the primary cell (by centroid or midpoint) |
| `secondary_cell_ids` | array[string] | Computed by cell assignment engine | Cell keys of all additional intersecting cells |
| `known_names` | array[string] | Extracted from source attributes | Human-readable identifiers for entity resolution |
| `source_authority` | string | Set by extractor from manifest | The organisation that published the source data |
| `source_dataset` | string | Set by extractor from manifest | Dataset identifier within the source authority |
| `source_feature_id` | string | Extracted from source feature | The feature's ID within its source dataset |
| `positional_accuracy_m` | float | Set by extractor from manifest or source metadata | Estimated positional accuracy in metres |
| `observation_date` | date | Extracted from source metadata where available | When the source data was captured (not when ingested) |
| `ingested_at` | datetime | Set at ingestion | When this entity entered the Harmony system |
| `fidelity_class` | string | Set by extractor from manifest | `structural` or `photorealistic` |
| `schema_version` | string | Set at ingestion | Entity schema version (e.g., `1.0`) |
| `geometry` | GeoJSON geometry | Extracted from source, normalised to WGS84 | The entity's spatial representation |

## Entity Type 1: `zoning_area`

**Source:** NSW Planning Portal — EPI Land Zoning (ArcGIS REST)
**Fidelity class:** structural
**Default resolution level floor:** r06 (large zoning polygons)
**Dedup strategy:** Source-system ID matching on `PCO_REF_KEY` or `SYM_CODE` + LGA combination

### Attribute Mapping

| Source Field (ArcGIS) | Harmony Field | Type | Required | Notes |
|---|---|---|---|---|
| `SYM_CODE` | `zone_code` | string | Yes | Standard zone code (e.g., R2, B1, E1, IN1, RU1) |
| `LAY_CLASS` | `zone_name` | string | Yes | Full zone name (e.g., "Low Density Residential", "General Industrial") |
| `EPI_NAME` | `epi_name` | string | Yes | The Environmental Planning Instrument name (e.g., "Central Coast Local Environmental Plan 2022") |
| `LGA_NAME` | `lga_name` | string | Yes | Local Government Area name |
| `PCO_REF_KEY` | `source_feature_id` | string | Yes | Unique string ID within the EPI dataset — primary dedup key. Confirmed as correct field by endpoint validation 2026-04-19. `OBJECTID` is also available but is numeric and may not be stable across service updates. |
| `SYM_CODE` | → `known_names` | string | Auto | Zone code added to known_names (e.g., "R2") |
| `LAY_CLASS` | → `known_names` | string | Auto | Zone name added to known_names |
| (geometry) | `geometry` | MultiPolygon | Yes | Zone boundary polygon |

### Source Metadata

| Field | Value |
|---|---|
| `source_authority` | `nsw_planning_portal` |
| `source_dataset` | `epi_land_zoning` |
| `positional_accuracy_m` | 5.0 (estimated — planning layers are not survey-grade) |
| `observation_date` | Use the EPI's commencement date if available; otherwise `null` |
| `fidelity_class` | `structural` |

### Notes

- Zone codes follow the NSW Standard Instrument LEP format. The `SYM_CODE` field is the primary human-readable identifier.
- Large zoning polygons (e.g., a rural RU1 zone covering several square kilometres) will span many cells. The geometry-adaptive resolution assignment will place these at higher levels (r04–r06). Smaller urban zones (a B1 commercial strip) will be assigned to r08–r10.
- The `epi_name` field is critical for provenance — it identifies which specific planning instrument defined this zone.
- **Pagination required:** The ArcGIS REST endpoint caps results at 1,000 per page (`exceededTransferLimit=true` observed). The adapter must paginate using `resultOffset` and `resultRecordCount`. Total feature count for Central Coast bounding box: 3,866 (3,718 within Central Coast LGA).
- **Endpoint validation note (2026-04-19):** `PCO_REF_KEY` (string) confirmed as the correct unique identifier via live endpoint validation. `OBJECTID` (numeric) is also present but `PCO_REF_KEY` is preferred as it is more likely to remain stable across service updates.

## Entity Type 2: `cadastral_lot`

**Source:** NSW Spatial Services — DCDB Lot dataset (ArcGIS REST at maps.six.nsw.gov.au, layer ID 9)
**Fidelity class:** structural
**Default resolution level floor:** r08 (parcel scale)
**Dedup strategy:** Source-system ID matching on lot + plan composite key derived from `planlabel`

### Attribute Mapping

| Source Field (ArcGIS REST) | Harmony Field | Type | Required | Notes |
|---|---|---|---|---|
| `cadid` | `cad_id` | string | Yes | Unique cadastral identifier |
| `lotnumber` | `lot_number` | string | Yes | Lot number within the plan |
| `sectionnumber` | `section_number` | string | No | Section number (may be null) |
| `planlabel` | `plan_label` | string | Yes | Combined lot/section/plan string (e.g., "1//DP123456"). This is the primary human-readable identifier. |
| (derived from `planlabel`) | `plan_number` | string | Yes | Extracted from `planlabel` — the numeric portion after the plan type prefix (e.g., "787786" from "DP787786"). |
| (derived from `planlabel`) | `plan_type` | string | Yes | Extracted from `planlabel` — the alphabetic prefix (e.g., "DP", "SP"). Note: `plantype` does not exist as a separate field in the live API. The `classsubtype` field offers a numeric alternative but `planlabel` parsing is more reliable. |
| `shape_Area` | `area_sqm` | float | No | Lot area in square metres. Note: field name is `shape_Area` (mixed case), not `shapestarea` as originally assumed. |
| `createdate` | `observation_date` | date | No | Date the lot was created in the DCDB |
| `lotnumber` + derived `plan_type` + derived `plan_number` | → `known_names` | string | Auto | Combined as "Lot {lot_number} {plan_type}{plan_number}" |
| `planlabel` | → `known_names` | string | Auto | The human-readable lot/plan label |
| (geometry) | `geometry` | MultiPolygon | Yes | Lot boundary polygon |

### Source Metadata

| Field | Value |
|---|---|
| `source_authority` | `nsw_spatial_services` |
| `source_dataset` | `dcdb_lots` |
| `positional_accuracy_m` | Varies: 0.2m (urban upgraded) to 12.5m (rural 1:25000 source). Use the `accuracy` field if present in source; otherwise default to 5.0. |
| `observation_date` | Use `createdate` from source if available |
| `fidelity_class` | `structural` |

### ArcGIS REST Access Notes

- **Endpoint:** `https://maps.six.nsw.gov.au/arcgis/rest/services/public/NSW_Cadastre/MapServer/9` (Lot layer)
- **Critical:** `inSR=4326` must be included in all bbox queries. The layer's native spatial reference is Web Mercator (102100). Without `inSR=4326`, the bbox coordinates are interpreted as Web Mercator and silently return 0 features.
- **WFS is not available.** Both WFS endpoints (`maps.six.nsw.gov.au` and `portal.spatial.nsw.gov.au`) returned 400 on GetCapabilities. WFS is not publicly exposed. ArcGIS REST is the only working programmatic access path.
- **Total features in Central Coast bounding box:** 133,943

### Dedup Composite Key

The natural deduplication key for NSW cadastral data is derived from `planlabel`: parse the plan type prefix and plan number, combine with `lot_number` and `section_number`. This uniquely identifies any parcel in NSW. The `cad_id` field is also unique but is an internal DCDB identifier that may not be stable across data releases.

**Dedup rule:** Match on composite key first. If a duplicate is found with different geometry, retain the version with the more recent `createdate`. Log the replacement.

## Entity Type 3: `building`

**Source:** OpenStreetMap (Overpass API)
**Fidelity class:** structural
**Default resolution level floor:** r10 (building scale)
**Dedup strategy:** Spatial proximity (15m threshold for centroid distance, same `building` tag value)

### Attribute Mapping

| Source Tag (OSM) | Harmony Field | Type | Required | Notes |
|---|---|---|---|---|
| `@id` (way ID) | `source_feature_id` | string | Yes | OSM way ID, prefixed with `osm:way/` |
| `building` | `building_type` | string | Yes | OSM building type tag (yes, residential, commercial, industrial, etc.) |
| `name` | `building_name` | string | No | Building name if tagged |
| `addr:street` | `address_street` | string | No | Street name |
| `addr:housenumber` | `address_number` | string | No | Street number |
| `addr:postcode` | `address_postcode` | string | No | Postcode |
| `addr:suburb` | `address_suburb` | string | No | Suburb name |
| `height` | `height_m` | float | No | Building height in metres (if tagged) |
| `building:levels` | `levels` | integer | No | Number of storeys (if tagged) |
| `roof:shape` | `roof_shape` | string | No | Roof shape classification |
| `start_date` | `observation_date` | date | No | Construction date if tagged |
| `name` | → `known_names` | string | Auto | Building name if present |
| `addr:street` + `addr:housenumber` | → `known_names` | string | Auto | Combined as "{number} {street}" if both present |
| (geometry) | `geometry` | Polygon | Yes | Building footprint polygon (converted from OSM way) |

### Source Metadata

| Field | Value |
|---|---|
| `source_authority` | `openstreetmap` |
| `source_dataset` | `osm_buildings_central_coast` |
| `positional_accuracy_m` | 3.0 (estimated — OSM contributor-traced from aerial imagery) |
| `observation_date` | Use `start_date` tag if present; otherwise `null` |
| `fidelity_class` | `structural` |

### OSM-Specific Processing Notes

- OSM buildings are represented as ways (closed polylines). The adapter must convert these to GeoJSON Polygons by closing the ring.
- Multipolygon relations (buildings with courtyards or complex shapes) require special handling — the relation members must be assembled into a MultiPolygon.
- The `building=yes` tag is the most common value and indicates a building of unspecified type. More specific values (`residential`, `commercial`, etc.) should be preserved in `building_type`.
- Height and levels data is sparsely tagged in Australian OSM. Expect <10% coverage for these fields.

## Entity Type 4: `road_segment`

**Source:** OpenStreetMap (Overpass API)
**Fidelity class:** structural
**Default resolution level floor:** r08 (road segments are typically longer than individual buildings)
**Dedup strategy:** Spatial proximity (5m threshold for Hausdorff distance) + same `highway` classification

### Attribute Mapping

| Source Tag (OSM) | Harmony Field | Type | Required | Notes |
|---|---|---|---|---|
| `@id` (way ID) | `source_feature_id` | string | Yes | OSM way ID, prefixed with `osm:way/` |
| `highway` | `road_class` | string | Yes | Road classification: motorway, trunk, primary, secondary, tertiary, residential, service, track, etc. |
| `name` | `road_name` | string | No | Road name |
| `ref` | `road_ref` | string | No | Road reference number (e.g., "A1", "M1") |
| `surface` | `surface_type` | string | No | Road surface: paved, asphalt, gravel, unpaved, etc. |
| `lanes` | `lane_count` | integer | No | Number of lanes |
| `maxspeed` | `speed_limit_kmh` | integer | No | Speed limit in km/h |
| `oneway` | `is_oneway` | boolean | No | Whether the road is one-way |
| `bridge` | `is_bridge` | boolean | No | Whether this segment is on a bridge |
| `tunnel` | `is_tunnel` | boolean | No | Whether this segment is in a tunnel |
| `lit` | `is_lit` | boolean | No | Whether the road is lit at night |
| `name` | → `known_names` | string | Auto | Road name if present |
| `ref` | → `known_names` | string | Auto | Road reference number if present |
| (geometry) | `geometry` | LineString | Yes | Road centreline geometry |

### Source Metadata

| Field | Value |
|---|---|
| `source_authority` | `openstreetmap` |
| `source_dataset` | `osm_roads_central_coast` |
| `positional_accuracy_m` | 3.0 (estimated — OSM contributor-traced from aerial imagery) |
| `observation_date` | `null` (OSM does not carry per-feature capture dates) |
| `fidelity_class` | `structural` |

### OSM-Specific Processing Notes

- Road segments in OSM can be very long (a highway spanning the entire Central Coast). The geometry-adaptive resolution assignment will handle this — long segments get assigned to higher-level cells (r06–r08) with many secondary cells.
- The `highway` tag is the primary classification. For the MVP, ingest all values from `motorway` through `residential`. Exclude `footway`, `cycleway`, `path`, and `steps` unless explicitly included in the manifest.
- The `oneway`, `bridge`, and `tunnel` tags use OSM boolean conventions: `yes`/`no` or `true`/`false`. Normalise to boolean during extraction.
- Speed limits in OSM may include units (`50 mph`, `60`). Assume km/h for Australian data. Parse and normalise to integer.

## Entity Schema Versioning

All entities carry a `schema_version` field set to `1.0` at ingestion. If the schema changes (new attributes added, types modified), the version increments. The pipeline must:

1. Record the schema version in the dataset registry alongside the ingestion run
2. Never break existing entities when the schema evolves — new fields are always additive
3. Entities ingested under schema 1.0 remain valid under schema 1.1+ (missing new fields default to `null`)

This versioning enables Pillar 4 to query entities by schema version and to apply backward-compatible enrichment to older entities without re-ingestion.
